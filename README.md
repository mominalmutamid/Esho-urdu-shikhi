# এসো উর্দু শিখি — Developer README

## Structure

```
esho-urdu-shikhi/
├── index.html       app shell (nav, view containers) — rarely changes
├── css/
│   └── style.css     full design system
├── js/
│   └── app.js          all app logic, fetches data/words.json
├── data/
│   └── words.json        THE FILE YOU'LL EDIT MOST — grows freely
├── manifest.json           PWA manifest, icons embedded as base64
└── sw.js                    service worker, network-first for everything
```

No `icons/` folder needed — both the manifest and the HTML favicon links
use base64-embedded icons directly, so there's nothing extra to place
anywhere.

## Why split files again (after going single-file for a while)

The single-file version existed to work around a real bug: GitHub's
**web drag-and-drop uploader** was unreliably flattening subfolders,
so `css/`, `js/`, and `data/` sometimes silently failed to upload.

That was never actually a "multiple files are bad" problem — it was an
"the tool I was using to upload is unreliable with folders" problem. The
fix is to stop using that uploader, not to avoid folders forever. With
proper git tooling, folders upload exactly as-is, every time.

## Recommended upload method: GitHub Desktop

1. Install [GitHub Desktop](https://desktop.github.com/) (free, no command
   line needed).
2. Clone your repo through it (File → Clone Repository).
3. Copy all the files above into the cloned folder on your computer,
   preserving the `css/`, `js/`, `data/` subfolders exactly as they are.
4. GitHub Desktop will show you every changed/added file in a list.
5. Type a short commit message, click **Commit to main**, then
   **Push origin**.

This never flattens folders, because it isn't a raw file drop — it's git,
doing what git actually does. From now on, growing `data/words.json` is
completely free: it never touches `index.html`, `css/`, or `js/`, and the
shell loads exactly as fast regardless of dictionary size.

(Command-line git works identically if you ever want it: `git add .`,
`git commit -m "..."`, `git push` — same reliability, more typing.)

## Running locally to test

```bash
cd esho-urdu-shikhi
python3 -m http.server 8000
# open http://localhost:8000/
```

Opening `index.html` directly via `file://` won't work — `fetch()` of
`data/words.json` is blocked by browser security on `file://` URLs.

## Extending the vocabulary — `data/words.json`

This is the only file you need to touch for ordinary vocabulary growth.
Every mode reads from the same `words` array. Per the file's own
`meta.schema_notes`, in short:

- **`id`** — keep stable once shipped; it's the localStorage join key for
  the library/streak history.
- **`urdu`** — must be correctly spelled; this is the ground truth the
  Spelling Mode and Sentence Mode validate against.
- **`example_sentence`** — `{urdu, bengali, pron_bn}`. Used by Study mode,
  Poetic mode, and Sentence Formation mode (word tiles are generated by
  splitting this sentence on spaces).
- **`is_poetic: true`** — pulls the word into Poetic Mode and the
  Word-of-the-Day rotation (`word_of_day_sequence`).
- **`type: "proverb"`** — the `urdu` field IS the full sentence (proverbs
  feed the "hard" tier of Sentence Formation Mode directly).

No app code changes needed for ordinary growth — only for a new game mode
or a new Visual-style feature.

## Service worker behavior

`sw.js` is network-first for every file in the app (the shell, the CSS,
the JS, and the dictionary). This means: whichever file you update and
push, visitors see the new version on their very next load while online,
with no cache-version bump required anywhere. It only falls back to the
cached copy when there's no network at all.

## Design & architecture notes

- **Palette**: night indigo / aged parchment / antique gold / pomegranate
  red / emerald — a manuscript aesthetic, not a generic app look. Tokens
  are CSS variables at the top of `style.css`.
- **Two Urdu typefaces**: Noto Nastaliq Urdu for display/poetic contexts,
  Noto Naskh Arabic for spelling tiles and study cards (clearer isolated
  letterforms).
- **Sentence Formation Mode** reuses `example_sentence` (easy tier) and
  proverbs (hard tier) — no separate sentence-bank needed; every sentence
  is one already verified elsewhere in the file.
- **Streak-free by design**: only "opened today or not" + a permanent
  library catalog are tracked — no consecutive-day counting or display.
